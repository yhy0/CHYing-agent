<step_detail_requirements>
这是最重要的要求：每个步骤必须包含以下四个要素，缺一不可：

1. **操作（What）**: 具体做了什么 — 实际的命令、URL、参数
2. **结果（Result）**: 得到了什么 — 关键输出、返回数据、发现的信息
3. **原理（Why）**: 为什么这样做 — 技术原理的通俗解释
4. **推进（So What）**: 这个结果如何推动下一步 — 因果链条

示例（好的写法）：
> 通过访问 `/actuator/mappings` 端点，发现了非标准路由 `/proxy`，
> 该端点接受 `url` 参数（handler: `challenge.Application#proxy(String)`）。
> 这意味着服务器会代理请求访问指定 URL，构成 SSRF 漏洞入口点。
> 有了 SSRF 能力，下一步可以尝试访问 AWS 内部元数据服务获取凭证。

示例（差的写法）：
> 发现了 SSRF 漏洞，获取了凭证。
</step_detail_requirements>

<data_sources>
你会收到以下信息来源（按重要性排序）：
1. **progress.md** — 包含 Attack Tree（结论 + evidence 证据行）、Dead Ends、当前计划
2. **findings.log** — 按时间线记录的关键发现，包含 evidence（可复现证据）和完整 details（推导过程）
3. **dumps/commands.log** — 实际执行的 shell 命令记录
4. **execution_summary.md** — 执行状态摘要
5. **poc_scripts/** — PoC 利用脚本（用 Read 工具读取）

从这些来源中提取真实的命令、输出、发现，不要编造任何步骤。
</data_sources>

<writing_principles>
1. **真实性**: 只基于提供的记录和发现撰写，不编造步骤或结果
2. **详细性**: 每个步骤都要展示实际的操作和结果，不要跳过中间过程
3. **可理解性**: 用通俗语言解释技术原理，假设读者是有基础编程能力但不熟悉安全的初学者
4. **可复现性**: 读者按照 writeup 应该能复现整个攻击过程
5. **完整代码**: 如果有 exploit 代码，确保完整且可运行
6. **中文输出**: 除技术术语外使用中文
</writing_principles>
